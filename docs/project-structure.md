# Project Structure

Use this guide for file placement questions. For platform-specific behavior and constraints, see [Architecture Notes](./architecture.md).

## Frontend Entry Point

`src/routes/+page.svelte` is the main Svelte page for the desktop UI. Keep page-level state, event handling, and composition here.

`src/routes/settings/+page.svelte` owns the settings page UI and delegates pure startup/update logic to frontend services.

## Frontend Service Helpers

`src/lib/services/` holds frontend service helpers used by the Svelte app. Put shared browser-side service logic here instead of growing `+page.svelte`.

Current high-traffic modules:

| Path | Responsibility |
| --- | --- |
| `workspace-state.ts` | `PageService` type, service add/edit/delete transforms, legacy startup migration. Must not import `workspace-groups.ts`. |
| `workspace-groups.ts` | Multi-workspace tree, workspace membership, active service, ordering. Pure transforms only. |
| `workspace-actions.ts` | Workspace transforms that also compute which webviews the caller should close/hide. No direct Tauri invokes. |
| `workspace-page-lifecycle.ts` | Startup state assembly, debounced storage writers, preload scheduling, activation-key/display-service helpers. |
| `webview-commands.ts` | Frontend wrappers for Tauri commands and the serialized webview command queue. |
| `service-runtime.ts` | Payload builder helpers and re-exported Tauri command payload types. Do not redefine those payloads elsewhere. |
| `service-editor.svelte.ts` | Add/edit dialog state and save flow. |
| `todo-panel.svelte.ts` | Todo notes state, mutations, panel open/close, and storage scheduling. |
| `drag-drop.svelte.ts` | Pointer drag state and handlers. |
| `runtime-badges.svelte.ts` | Runtime unread badge map and badge event application. |
| `settings-page-state.ts` | Settings page startup and workspace color helpers. |
| `updater.ts` | Wrapper around `@tauri-apps/plugin-updater` for settings UI tests and update flow. |

Generated Tauri command wrappers live in `src/lib/tauri-commands.ts`. This file is committed but generated by `tauri-specta`; do not edit it manually.

## Rust Backend Layout

`src-tauri/src/main.rs` is the executable entrypoint.

`src-tauri/src/lib.rs` is the main Tauri command and app wiring module. Keep it focused on wiring, and extract backend logic into sibling Rust modules.

Current Rust modules:

| File | Responsibility |
| --- | --- |
| `app_state.rs` | Shared managed state such as active webview, badge prefs, audio mute state, panel width, and resize generation. |
| `badge_payload.rs` | Badge payload parsing and normalization. |
| `desktop_ui.rs` | Native tray icon and context menu helpers. |
| `download_dialog.rs` | Native download handling for service webviews. |
| `file_drop.rs` | File drop forwarding into active service webviews. |
| `macos_defaults.rs` | macOS runtime defaults such as press-and-hold key repeat behavior. |
| `navigation_bridge.rs` | Intercepts `ferx.notify`, `ferx.resource`, shortcut, and download navigation URLs from remote webviews. |
| `service_runtime.rs` | URL parsing and service classification. |
| `service_storage.rs` | Service storage key to data-store/session-directory mapping. |
| `service_webview.rs` | Service webview setup and initialization script assembly. |
| `service_webview_badge_scripts.rs` | Rust loaders/composition for badge engine scripts in `src-tauri/scripts/`. |
| `service_webview_resource_usage.rs` | Rust loader for the resource-usage monitor script. |
| `service_webview_runtime_scripts.rs` | Runtime script loaders for common webview behavior, notifications, spellcheck, auth compatibility, and audio mute. |
| `webview_commands.rs` | Tauri commands for service webview lifecycle and native webview side effects. |
| `window_layout.rs` | Physical-pixel layout calculations. |
| `lib_tests.rs` | Rust unit tests. Prefer behavior tests over source-string assertions. |

Injected JavaScript loaded into service webviews lives under `src-tauri/scripts/`. Rust includes those files at compile time with `include_str!()`.

## Where New Logic Should Go

Put new frontend service logic in `src/lib/services/` when it runs in the Svelte app and can be reused across page code.

Put backend service or webview logic in `src-tauri/src/` when it belongs to the Rust side of the app. Keep `lib.rs` as wiring and place focused implementations in separate Rust modules.

Put injected webview JavaScript in `src-tauri/scripts/` and expose it through a narrow Rust loader when the script is large enough to be maintained outside a Rust string literal.
